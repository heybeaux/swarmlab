## 2. The open question: what verifies a claim in the ground store?

Spec 18 proved the **audit architecture** closes the fabrication hole *when a ground store
exists*. But the store in the retest was a **seeded oracle** — we told it the truth in advance.
That's fine for proving the mechanism; it is not a deployment story.

**Settled:** Engram is the ground store. It already has the right substrate — content-addressed
digests, versioned facts, anti-entropy healing (spec 16), provenance per fact. A fact in Engram
can prove *it hasn't been mutated* and *who wrote it*.

**Open:** what mechanism gets a claim *into* the store as `supported` in the first place —
i.e., what does verification actually rest on? Candidate mechanisms, none yet tested:

1. **Provenance-chain verification** — a claim is `supported` iff it traces to a signed
   observation event (AOP/Sonder chain) from an agent that *directly executed or measured* the
   thing. Verification = replay the chain. Strong for operational facts ("tests passed",
   "deploy succeeded"); useless for world knowledge.
2. **Retrieval-grounded verification** — claim must cite a retrievable source; verifier
   fetches and checks entailment. Strong for documented knowledge; inherits the retrieval
   corpus's biases and staleness.
3. **Cross-model adversarial verification** — a Parliament panel with *different* models
   fact-checks the claim under the pinned criterion. Turns spec 15/18 machinery on the store's
   own inputs. Catches fabrication styles a single model repeats; costs tokens; can converge on
   shared model errors.
4. **Human attestation** — Beaux signs a fact. Gold standard, doesn't scale; should be the
   root of trust the other three chain up to, not the everyday path.

**Likely answer is a hierarchy** (attestation > provenance > retrieval > cross-model), with the
verification *tier* recorded on the fact itself so the audit can weight `supported-by-provenance`
above `supported-by-cross-model`. **Recommended next step:** spec a small Engram extension —
`verification_tier` + verifier identity on `VersionedFact` — then an experiment where liars
attack each tier. Do NOT bolt this into spec 18's `TableFactStore`; that stays an oracle for
harness use.

## 3. Proposed experiment: exp-14 — Delegation Decay & Trust Routing

**Not covered by existing experiments.** exp-01 (telephone-compiler) and exp-05 (bug-telephone)
measure *content* degradation through a fixed relay chain — nobody chooses whom to delegate to,
and no agent is incapable. Delegation *choice*, capability *asymmetry*, and *trust formation*
are unmeasured. This fills that hole.

### Question A — How does a task deteriorate as it's decomposed and delegated?

**Setup:** a root agent receives a task with N verifiable requirements (a spec with machine-
checkable acceptance criteria — e.g., a config-generation task where every requirement is
assertable). Root decomposes and delegates subtasks to workers; workers may sub-delegate.
Sweep **delegation depth** (0–4) and **branching factor** (1–4). Same task, same seeds, every
condition.

**Measures:**
- `requirementSurvival(d)` — fraction of the N criteria still satisfied in the assembled
  result, per delegation depth. The delegation analogue of exp-08's fidelity curve.
- `intentDrift` — requirements *reinterpreted* (present but wrong) vs *dropped* (absent).
  Distinct failure modes; exp-01 says reinterpretation dominates.
- `integrationTax` — failures introduced at reassembly, not inside any subtask (interface
  misunderstandings between siblings — this is where exp-12's false-friend result predicts
  the bleeding).
- `costAmplification` — total tokens vs the depth-0 baseline doing the whole task alone.

**Hypothesis:** requirement survival decays roughly exponentially in depth, and the dominant
loss at depth ≥ 2 is integration mismatch, not subtask failure — i.e., delegation fails at the
*seams*.

### Question B — Does an agent without memory re-delegate to an incapable agent forever?

**Setup:** a worker pool with a planted **incapable agent** — plausible on paper (name,
self-description), but it reliably fails a specific subtask class (fails cleanly, or worse,
returns confident wrong output — both arms). The root agent runs R sequential rounds of similar
tasks and chooses delegates each round.

Three arms, same seeds:
1. **Amnesiac** — root has no memory of past rounds (fresh context each round). *Hypothesis:
   selection rate for the incapable agent stays flat at chance for all R rounds — the agent
   pays the same failure tax every round, forever.*
2. **In-context memory** — root keeps a transcript of past outcomes in its own context.
   Predicted to converge but degrade as R outgrows the window (and it dies with the session —
   that's the point).
3. **Engram-backed** — root writes a **capability observation** per delegation outcome
   (`agent X, task class Y, outcome Z`, provenance-tiered per §2) and queries before choosing.
   *Hypothesis: selection of the incapable agent → ~0 within a handful of rounds, persists
   across session resets (kill the root between rounds — arm 3 is the only one that survives),
   and transfers: a **different** root agent reading the same store avoids the incapable agent
   on round 1.*

**Measures:** re-delegation rate to the incapable agent per round; rounds-to-convergence;
wasted tokens on doomed delegations; cross-session persistence (the reset test); cross-agent
transfer (the second-root test). The transfer result is the Engram thesis in one number:
**trust is a fact, and facts belong in a store, not a context window.**

**The confident-wrong arm matters most:** an incapable agent that *fails loudly* is cheap to
learn; one that returns plausible garbage poisons downstream integration (ties back to
Question A's `integrationTax`) and is exactly what capability observations + fact-checking
(spec 18) should catch.

## 3b. Two follow-on specs written — builders can dispatch directly

exp-14 surfaced two findings that DO demand stack work; both are now specs, no
further design needed:

- **`specs/20-trust-forgiveness.md` (exp-15).** exp-14's exclusion rule
  (`experiments/14-delegation-decay/src/trust.ts:126`, `failures <= successes`) is
  unforgiving — it permanently benches ~0.22 capable workers/trial that hit an early
  transient failure. Spec 20 tests forgiveness policies (time-decay vs
  probation/retry-quarantine) that recover capable workers **without** re-opening the
  door to the incapable delegate. Inherits exp-14 Part B seeds/pool for a fair
  cell-for-cell control. Deliverable: `capableExcluded → 0` held against
  incapable-selection ≤ 0.05, plus a stack recommendation for the real trust router.
- **`specs/21-handoff-requirement-guards.md` (exp-16).** exp-14 Part A refuted the
  seam hypothesis — the dominant loss is silent omission (dropped requirements).
  Spec 21 tests a handoff requirement-manifest guard (presence tier vs value-echo
  tier) that catches drops at the seam that introduced them. Reuses exp-14 Part A
  task/decay/scoring for a fair control. Deliverable: deep-cell survival recovery vs
  guard token cost, plus where in the real delegation path (sonder/lattice handoff or
  AOP payload contract) the manifest should live.

Both follow specs 14–19 rules (real packages `file:`-linked never reimplemented,
seeded/deterministic/replay-verified, honest numbers even if red). Spec 20 is the
more directly shippable — it's a concrete policy on the reconciliation trust layer.
